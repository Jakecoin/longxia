# 项目蓝图

最后更新：2026-03-04

## 目的

Longxia 是一个“受控边界”AI 网关分支。它不是为了保留上游的全部能力，而是为了把现有代码库收敛成一个更容易部署、更容易运营、也更适合国内环境落地的产品底座。本文件的作用，是给后续开发提供稳定的产品蓝图，避免功能继续无边界扩张。

## 产品定义

Longxia 需要同时被理解为三件事：

- 一个给个人或团队自部署的受控网关
- 一个面向未来中国市场 SaaS 的基础平台
- 一个以“代码审核 + 构建期集成”为主，而不是“运行时任意扩展”为主的硬分叉版本

因此，Longxia 不应再被定义成“什么都能接”的通用连接器集合。它应该被定义成“少量可信核心 + 审核制扩展”的运营型平台。

## 不可妥协的产品规则

### 1. 出网边界必须受控

- 运行时出网默认走 allowlist。
- 新的外部服务接入必须经过代码评审与显式批准。
- 安装和升级必须优先使用经过批准的镜像和受控源码地址。

### 2. 不提供公共运行时扩展体系

- `install`、`update`、`enable`、`disable` 这类运行时插件管理，不应作为标准用户能力继续存在。
- 可选能力应优先在构建期打包。
- 开发阶段可以保留本地扩展加载，但它不应成为正式产品分发面。

### 3. SaaS 默认是多租户，而不是一用户一台机器

- 共享 SaaS 是默认商业形态。
- 每用户独立基础设施只能作为付费升级层。
- `tenant_id` 必须贯穿请求、计费、审计和工具执行。

### 4. 分发链路必须适配国内环境

- 核心安装文档和脚本要优先走国内可访问镜像。
- 首次使用不能依赖被阻断的公网页面。
- 任何必须依赖的公网组件，都要有国内可用的替代方案或兜底路径。

## 产品模块

### 模块 A：Gateway 核心

Gateway 仍然是系统的主控制点和策略执行点。

职责：

- 维护操作员与节点会话
- 执行认证与配对
- 负责请求路由
- 统一接入审计与用量统计
- 向 CLI、Web UI、移动端暴露稳定控制接口

后续租户策略、出网策略、队列策略都应尽量收敛到 Gateway 执行，而不是散落在各处。

### 模块 B：客户端表面

客户端包括：

- CLI
- Web 控制台
- macOS / iOS / Android 客户端

职责：

- 配置与初始化
- 运营可见性
- 受控调用 Gateway 能力
- 本地诊断与支持入口

客户端不应再引入隐藏的外部依赖，也不应提供绕过受控安装模型的“快捷入口”。

### 模块 C：工具与动作执行层

工具能力需要明确分成两类：

- 可信内建能力
- 需要沙箱的高风险能力

设计原则：

- 风险可控、爆炸半径小的能力，可以继续留在主进程
- 有执行风险、计费需求或可能承载用户自定义数据的能力，应逐步迁移到短生命周期沙箱任务

这会直接决定后续的隔离、审计和商业化能力。

### 模块 D：SaaS 控制平面

这个模块当前未完整实现，但项目设计必须提前为它留出接口。

职责：

- 租户管理
- 认证配置档案
- 密钥与凭据元数据
- 用量统计
- 计费事件
- 审计与支持视图

即使当前主要是自部署版本，代码结构也应逐步为这层能力留下清晰的扩展缝隙。

## 部署模型

### 共享 SaaS（默认）

目标用户：

- 个人用户
- 小团队
- 早期付费客户

技术形态：

- 共享控制平面
- 一个或多个共享 worker 池
- 租户级配置、鉴权、计费
- 逻辑隔离 + 沙箱执行

为什么这是默认方案：

- 成本最低
- 迭代最快
- 最符合低到中等客单价市场

### 专属资源池（Pro）

目标用户：

- 对性能隔离有要求的团队
- 并发更高或出网策略更严格的客户

技术形态：

- 专属队列分区
- 专属 worker 分区
- 可选专属代理出口
- 不做全量基础设施拆分，但提供明显更强的隔离感

这是最主要的中档付费升级路径。

### 专属部署（Enterprise）

目标用户：

- 企业客户
- 合规场景
- 需要私网接入的客户

技术形态：

- 独立 namespace、集群或 VPC
- 客户级密钥与出网边界
- 更严格的审计与运维保障

这应被视为单独的企业产品，而不是默认形态。

## 系统设计原则

### 身份必须全链路存在

每一次请求都应该能够被追溯到：

- 租户
- 操作主体
- 设备
- 智能体或执行上下文

这是后续计费、支持、排障和风控的基础。

### 构建期优先于运行时变更

当在下面两种方式之间选择时：

- 运行时下载/安装能力
- 构建期打包能力

Longxia 默认应选择后者。只有在运维收益明确大于风险时，才考虑放宽。

### 高风险执行优先短生命周期

长期常驻进程适合 Gateway 和稳定控制服务；高风险动作应优先使用短生命周期执行单元，并带上资源限制、超时和清理策略。

### 兼容性不等于产品方向

仓库里保留部分上游命名是可以接受的，但这不应影响 Longxia 的产品方向判断。兼容层只是过渡成本，不是路线约束。

## 近期技术路线

### Phase 1：边界与分发收敛

目标：

- 去掉被阻断或脆弱的上游外链
- 锁死运行时扩展入口
- 将安装、升级、帮助链路统一收敛到受控路径与国内镜像

完成标准：

- 默认安装/升级使用受控镜像
- 被移除渠道不再出现在正常产品表面
- 上游文档不再定义主用户路径

### Phase 2：为多租户做内核准备

目标：

- 即使在自部署模式下，也让核心请求链路具备“租户可感知”能力

必须具备的结构：

- `tenant_id` 透传
- 每动作用量统计
- 凭据作用域模型
- 审计事件模型
- 沙箱调度抽象

完成标准：

- 核心请求路径能够稳定归属、计量、审计

### Phase 3：托管 SaaS 基线

目标：

- 在当前核心之上搭建最小可运营的托管服务

需要的组成：

- 托管鉴权
- 计费与配额
- worker 调度
- 支持后台
- 共享层的部署模板

完成标准：

- 在不按客户拆机器的前提下，能够运行共享多租户版本

### Phase 4：高级隔离套餐

目标：

- 在不重写核心的前提下，提供可售卖的更强隔离能力

需要的能力：

- 队列分区
- worker 分区
- 专属出网配置
- 可选 namespace / 集群隔离

完成标准：

- Pro 与 Enterprise 成为稳定产品能力，而不是临时运维项目

## 对仓库演进的约束

后续评审代码时，应默认执行这些判断：

- 拒绝悄悄扩大出网面的新增能力
- 拒绝重新引入公共插件市场
- 优先把策略集中到统一模块，而不是新增零散例外判断
- 优先写清晰的 Longxia 文档，而不是继续默认继承上游假设

这个分支应该越来越容易理解，而不是越来越像上游的宽口径集合。

## 决策清单

在增加任何新能力前，先回答：

1. 它是否扩大了默认出网范围？
2. 它能否在构建期打包，而不是运行时安装？
3. 它是否符合共享 SaaS 这一默认形态？
4. 它是否保留了升级到更强隔离层的路径？
5. 它是否还能被“受控边界”模型清楚解释？

如果第 5 个问题回答不了，这个功能大概率就不属于 Longxia 的默认产品。
